# Investigative Probe (정적으로 안 갈리면 런타임으로 본다)

핵심 명제 (이 한 줄이 전부):
> **코드만으로 어느 후보가 진짜인지 모르면, 그 값을 *실제로 실행시켜 관찰*하면 안다.**
> 정적 반증은 가설을 *줄이고*, 런타임 관찰은 가설을 *확정*한다.

**언제**: 정적 분석(예: root-cause-lifecycle) 후에도 **후보가 2개 이상 남거나** 반증이 흔들릴 때.
"이 값이 *이 지점에서 실제로* stale/틀린 값인가?"를 코드만으로 단정 못 할 때.
**아닐 때**: 코드만으로 이미 확실하면 프로브 낭비 — 만들지 마라. 또 **수정의 재현 테스트가 아니다**
(그건 `reproduce-first`; 거긴 정답 오라클로 *커밋·유지*, 여긴 *조사용 버리는 코드*로 *원복*).

## 재현 테스트 vs 조사 프로브 (혼동 금지)

| | reproduce-first 재현 테스트 | investigative probe |
|---|---|---|
| 목적 | *증상*이 존재함을 증명(RED→GREEN 오라클) | *메커니즘/후보*를 런타임 값으로 판별 |
| 수명 | 커밋·영구 보존(불가침 오라클) | **throwaway** — 관찰 후 원복(커밋 안 함) |
| 보는 것 | 외부 동작(거부/통과) | **내부 값**(필드/캐시/변수의 런타임 상태) |
| 프로덕션 코드 | 안 건드림 | 안 건드림(계측=임시 로그만, 동작 불변) |

## 절차

1. **판별 질문을 명확히** — 경쟁 후보 A vs B가 갈리는 **단 하나의 관측**은? "지점 P에서 값 V가
   A면 X, B면 Y로 다르다." 이 *판별변수*가 없으면 프로브가 의미 없다 — 정적으로 더 좁혀라.
2. **최소 시나리오 구동** — 증상을 일으키는 **가장 작은 setup**을 코드로(예: go test). 시간적 시퀀스가
   원인이면 그 시퀀스(트리거 → 후속 이벤트/유휴 구간)를 그대로 재현.
3. **후보 지점 계측** — 각 후보 지점 P에서 의심 값 V를 **임시로 노출**(`t.Logf`/임시 로그 라인/덤프).
   프로덕션 *로직*은 바꾸지 마라(관찰만 추가).
4. **실행·관찰** — `go test -run <probe> -v`(동시성 의심 시 `-race`). 출력에서 V의 실제 값을 읽는다.
5. **판별** — 예측한 stale/틀린 값이 *실제로* 나타나는 후보 = 진짜. 나머지는 **관찰로 반증**(정적 반증보다 강함).
   여전히 안 갈리면: 판별변수를 다시 골라 2로, 또는 e2e/로그(chainbench)로 격상.
6. **원복 + 기록** — 계측·프로브 테스트를 제거(커밋 금지). 관찰값을 진단(diagnosis.md / analysis.md
   Root cause)에 **증거로** 남긴다("P에서 V=… 관측 → 후보 A 확정, B 반증").

## 경계
- **버리는 코드**: scratch 파일/임시 함수. 끝나면 원복 — 데이터셋·브랜치에 남기지 않는다.
- **관찰만**: 프로덕션 동작을 바꾸는 계측 금지(임시 로그/assert만). 동작을 바꾸면 그건 더 이상 관찰이 아니다.
- **최소·결정적·시간상자**: 가장 작은 재현, 비결정성 제거, 한두 번 돌려 답이 안 나오면 정적/입력으로 복귀.
- diagnose 모드(read-only)에서도 **프로브는 허용**(throwaway·원복이므로) — 단 프로덕션 수정·브랜치·전이는 여전히 금지.

## 워크드 예 (방법 예시일 뿐, 규칙 아님)

후보 둘: (A) 한 갱신 경로의 race로 값이 일시 불일치, (B) 어떤 가드가 조건이 같으면 갱신을 skip해 stale.
- 판별변수: 증상 발생 지점에서 "갱신 대상 값"이 (A)면 *간헐적으로* 옛값, (B)면 *유휴 구간 내내* 옛값.
- 프로브: 트리거 후 **추가 활동 없이 유휴를 유지**하는 최소 시나리오를 돌리고, 거부 지점에서 두 후보 값에
  `t.Logf`. 관찰: 유휴 내내 일관되게 옛값 → **(B) 가드 확정**, (A) race 반증(간헐적이지 않으므로).
- 원복하고 "유휴 구간 내내 stale = 가드 경로" 를 진단 근거로 기록.

## 두 변종 — 재현 티어를 따른다 (in-process / e2e)

프로브는 재현 티어와 같은 티어로 돈다:

- **in-process (simulation)**: 위 절차 그대로 — 버리는 go test에서 후보 값을 `t.Logf`로 찍어 관찰.
  단일 프로세스로 충실히 재현되는 증상(순수 함수, 인코딩, gas/fee, 단일 노드 상태전이)일 때.

- **e2e (바이너리 계측 — chainbench)**: 단일 프로세스로 재현 못 하는 증상(합의·txpool·멀티노드·
  거버넌스·노드 간 발산 등)은 **프로덕션 바이너리에 임시 로그를 박아** 관찰한다. analyzer **§5c
  diagnosis loop**가 이 변종의 운영 절차다:
  1. 후보 지점 P에 임시 로그 라인(`log.Info/Warn` 또는 stderr `fmt.Fprintf`) 추가 — **관찰만,
     프로덕션 로직/동작 불변**.
  2. `make gstable` 재빌드 → `chainbench_restart({binary_path, project_root})`로 **같은 네트워크에
     새 바이너리만 스왑**(매 iter마다 init 새로 하지 말 것 — 느림).
  3. **재현 테스트만 1회 재실행**(`chainbench_test_run repro/<...>`). 3×반복·회귀·parent 재빌드는
     여기서 하지 않는다(그건 evaluator의 최종 확인이지 진단이 아님). 1회면 계측 로그가 나온다.
  4. `chainbench_log_search` / `log_timeline` / `failure_context` / `node_rpc` / `txpool_inspect`
     로 계측 라인의 **실제 값**을 읽는다.
  5. 예측값이 실제로 나오는 후보 = 확정, 나머지는 **관찰로 반증**. 안 갈리면 값 생애주기를 한 hop
     더 깊이 계측하고 2로.
  6. **원복**: `git -C {repo_root} checkout -- <files>` 로 계측 제거 — **절대 fix 브랜치/PR에 남기지
     말 것**(Implementer는 깨끗한 트리에서 시작). 재현 오라클 `.sh`는 유지. 끝나면 `chainbench_stop`.
  7. 관찰값을 `analysis.md`의 `## Root cause`(broken edge `file:line` + 관측 로그 한 줄)와
     `affected_sites`(관찰이 드러낸 sibling 경로 포함)에 **증거로** 남긴다.

  비용: iter당 재빌드 ~1–2분 + 재시작 + 재현 1회 — 풀 IMPLEMENT→EVALUATION 바운스(~30분 e2e)보다
  훨씬 싸다. **시간상자 ≤3–4 iter**; 못 갈리면 최선 가설 + 남은 판별변수를 기록하고 넘긴다.
