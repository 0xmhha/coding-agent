# Root-Cause Lifecycle Interrogation (진단 추론 — L2→근본원인 다리)

핵심 명제 (이 한 줄이 전부):
> **버그 = "소비자가 쓰는 값 == 생산자가 마지막으로 쓴 값" 불변식이 *어느 edge*에서 깨진 것.**
> 값의 모든 edge를 빠짐없이 열거하면 깨진 곳을 못 놓친다.

**언제**: "값이 틀리다 / stale 하다 / 변경이 반영 안 된다" 류 버그 진단(diagnose, bugfix 사이클).
**아닐 때**: 새 기능 설계(→ planner §5.2b), 프로토콜 모양·합의 안전성 판정(→ stablenet-invariants).
**관계**: cks를 대체하지 않는다. cks가 *후보 코드*를 주면, 이 절차가 *인과 사슬*로 조립한다.

## 절차 (이 순서를 강제한다)

1. **후보 값(들)을 지목** — 증상이 *무엇에 관한* 값인가? 필드/슬롯/변수 단위로 구체적으로. 보통 1개지만,
   정적으로 여러 값이 의심되면 **복수 후보를 유지하라(조기에 하나로 줄이지 마라)**.
2. **생애주기를 그린다** — `produce → store/copy → consume`. **모든 복사본·캐시를 빠짐없이** 나열.
   cks 매핑: `find_callers`/`impact_analysis` = 복사본·소비자 열거, `change_history` = 생산자 변경 시점.
3. **edge별 실패모드를 열거** (보편 분류 — 도메인 무관) —
   - produce: 안 씀 / 잘못 씀 / 늦게 씀(타이밍)
   - store: **소스가 바뀌어도 갱신 안 됨(stale)** / 키 틀림 / 복사본이 갈라짐
   - consume: 잘못된 소스를 읽음 / stale 복사본을 읽음 / 비교 로직 오류 / 전제조건 틀림 / 누락된 검증·가드
4. **각 실패모드 → 함수 1개로 매핑** — 추측 말고 `file:line`. 없으면 cks로 찾는다.
5. ⭐ **시간적 시퀀스 추적** — 증상이 **"트리거 X 이후 한동안 지속되다 사라진다"** 류이면, X *한 시점*이 아니라
   **X 이후 이벤트 시퀀스**(후속 호출·이벤트·주기적 갱신)를 따라가라. 특히 ⭐ **증상을 *해소*시키는 이벤트를
   찾아라** — 그 이벤트가 갱신/재계산하는 대상이 곧 **누락된 갱신 지점**이다. (정적 그래프는 한 시점의 연결만
   보여준다. 트리거 *직후*가 아니라 그 *뒤*에 무엇이 다시 실행되어 값을 고치는지는 직접 추론해야 한다.)
6. ⭐ **증상의 구별특징으로 반증** — 예 "한 방향만 stuck", "특정 주체만 영향", "특정 조건에서만". 그 특징을
   *설명하지 못하는* 가설은 약화한다. **단, 정적 반증은 가설을 *줄일* 뿐 *확정*하지 못한다** → 7·9.
7. ⭐ **소스까지 역추적** — stale 값을 찾아도 멈추지 마라. "이 값의 생산자는?"을 깨진 edge나 생산자에 닿을
   때까지 반복. **첫 캐시는 보통 *증상*이지 원인이 아니다.**
8. **캐시 무효화 점검** — 모든 캐시마다 "소스가 바뀔 때 무효화/갱신하는 코드가 있나?".
   **invalidator 0 인 캐시 = 유력 용의자.**
9. ⭐ **다중-후보 판별(런타임)** — 위를 다 했는데도 **후보가 2개 이상 남거나** 반증이 흔들리면(정적 코드만으론
   어느 후보가 실제로 fire하는지 모르면), **`investigative-probe` 스킬**로 증상 시나리오를 구동하며 각 후보
   지점의 런타임 값을 관찰해 가린다. **판별 전에는 단일 원인으로 확정하지 말고** competing으로 보고하라.

> ⭐ 5·7·9는 **검색·열거만으로는 안 나오는, 가장 자주 건너뛰는 세 동작**이다.
> 5를 빠뜨리면 트리거 시점만 보고 "그 다음"을 놓치고, 7을 빠뜨리면 첫 stale 복사본(증상)에서 멈추며,
> 9를 빠뜨리면 후보가 갈릴 때 *추측*으로 단일 확정해 틀린다 — 부분진단·오진의 전형이다.

## 산출 (diagnosis.md / plan-fix-N.md 의 Root cause 절에 반영)

- 지목한 값(들) + 생애주기: producer / 복사본 목록 / consumer
- **깨진 edge + `file:line`** (런타임으로 확인된 것 우선; 미확인 competing 후보가 있으면 그대로 표기)
- 반증으로 탈락시킨 경쟁 가설(왜 아닌지 한 줄씩) + 런타임 프로브로 판별했다면 그 관측값
- 확신도 + 무엇이 확신을 올리나(어떤 *판별변수*를 어떤 관측/재현이 보여주는가)

## 워크드 예 — PR77 (압축)

> 아래는 *방법을 보여주는 예시일 뿐, 규칙이 아니다*(도메인 단서를 절차에 넣지 말 것).

값 = effective minTip. 복사본 = ① state 슬롯 ② `AnzeonTipEnv.currentBlock` ③ `Transaction.anzeonTipCap` ④ `pool.gasTip`.
- 스텝8: ③ invalidator 0 → stale. **첫 캐시 = 증상.**
- 스텝5(시간적): "증상이 *다음 활동(non-empty 블록)* 때 사라진다" → 그 이벤트가 부르는 갱신 경로(`SetCurrentBlock`)가 곧 누락 지점. ← 트리거 시점만 보면 놓침.
- 스텝7 역추적: ③의 소스 = ②(`GetAnzeonTipCap`) → ②의 store = `SetCurrentBlock`이 동일-조건이면 갱신 skip → **진짜 source.**
- 스텝9(프로브): 정적으로 후보가 갈리면(예: 다른 갱신 경로 race vs 위 가드) — 증상 시나리오를 구동해 **어느 값이 stale인지 런타임 관찰**로 확정.

→ 첫 캐시에서 멈추거나(스텝7 누락), 트리거 시점만 보거나(스텝5 누락), 후보 갈릴 때 추측하면(스텝9 누락) 오진. **5·7·9를 끝까지** 돌려야 진짜 원인에 닿는다.
