# [CKG] 그래프 기반 코드 검색 효과 측정 — 4-Way 비교 결과 (v6)

## 배경 / 목적
Code Knowledge Graph(ckg)가 AI 코드 이해에 실제로 효과가 있는지를 **느낌이 아닌 수치**로 검증한다.
미리 정답을 아는 **30개 코드 질문**(go-stablenet 12개 도메인)을 **4가지 컨텍스트 제공 방식**으로 제출하고 응답 품질을 자동 측정·비교했다.

- 인덱스: go-stablenet `dev`(`c051d50b`) 기준, **`make gstable` 빌드 파일만** 스코핑(미사용 합의엔진 제외, Solidity 컨트랙트 포함)
- 검색 3회 평균 / LLM 판정(Sonnet, 3회 전부 = 360셀, PARSE_FAIL 0) / 입력은 키워드·문장만(정답은 채점 전용 — 누수 없음)
- **v6 변경점**: cks 검색 도구(semantic_search/search_text/find_symbol/find_callers/find_callees/get_subgraph)에 **`exclude_tests` 쿼리 필터** 추가. 공유 분류기(`pkg/testpath.IsTest`)가 테스트 파일 + 테스트 전용 헬퍼(`testutil*.go`, `test/` `testdata/` 디렉토리 등)를 쿼리 시점에 제거. β/γ는 composer를 우회해 그래프를 직접 호출하므로, 이 필터로 비로소 테스트가 걸러진다. composer의 test-demotion도 같은 분류기를 쓰도록 통일.
- (v5 유지) glossary 쿼리 확장으로 δ 정답존재 83%→97%. (v4 유지) precision 지표 분리.

## 비교 방식
| 코드 | 방식 |
|------|------|
| α | 파일 원문 grep (기준선, 그래프 미사용) |
| β | 그래프 전체 + 노드 본문 (`exclude_tests`) |
| γ | 필요한 정보 개별 조회 + 노드 본문 (`exclude_tests`) |
| δ | 자동 선별 (get_for_task, glossary 확장) |

## 핵심 결과
| 방식 | 정답존재 | 정답집중도 | 관련성precision | 설계충분성 | 평균토큰 | 효율 | 테스트오염 |
|------|------:|----------:|----------:|----------:|--------:|------:|----------:|
| α grep | 70% | 0.266 | 0.776 | 82% | 10,829 | 7.59 | 15/90 |
| β 그래프전체 | 80% | 0.040 | 0.328 | 80% | 8,615 | 9.29 | **0/90** |
| γ 개별조회 | 83% | 0.072 | 0.410 | 82% | 5,972 | 13.77 | **0/90** |
| **δ 자동선별** | **97%** | 0.263 | **0.651** | **87%** | **4,800** | **18.06** | **0/90** |

- **정답존재**(위치정확도): 정답 파일을 검색 결과에 포함했는지
- **정답집중도**(strict): 가져온 파일 중 정확히 정답 파일의 비율
- **관련성precision**(LLM): 정답 + 설계에 유용한 맥락의 비율
- **설계충분성**(LLM): 그 컨텍스트로 기능 수정/설계가 가능한지
- **테스트오염**: 검색 결과에 테스트/테스트전용 파일이 1개 이상 섞인 셀 수(90=30문항×3회). 판정식은 `pkg/testpath.IsTest`와 동일

## v5 → v6 변화 (exclude_tests 필터 효과)
| 지표 | β | γ | δ |
|------|:-:|:-:|:-:|
| **테스트오염** | 78/90 → **0/90** | 69/90 → **0/90** | 3/90 → **0/90** |
| 평균토큰 | 11,811 → **8,615** | 7,056 → **5,972** | 4,767 → 4,800 |
| 관련성precision | 0.293 → **0.328** | 0.372 → **0.410** | 0.641 → 0.651 |
| 효율 | 6.77 → **9.29** | 11.34 → **13.77** | 18.18 → 18.06 |
| 정답존재 | 83% → 80% | 83% → 83% | 97% → 97% |

쿼리 단계에서 테스트를 거르니 그래프 직접 호출(β/γ)의 테스트오염이 전부 사라졌고, 테스트 본문이 주입되지 않아 **토큰이 감소(β −27%, γ −15%)하면서 관련성precision·효율은 오히려 상승**했다 — β/γ가 끌어오던 테스트는 유용한 맥락이 아니라 노이즈였다는 뜻. δ는 이미 깨끗(3/90, 옛 분류기가 놓친 헬퍼 1종)했고 공유 분류기로 0이 됨. 유일한 회귀는 β 정답존재(83→80%): 테스트 hit이 top-k에서 빠지며 β가 확장하는 심볼 시드가 바뀌어 gov 도메인 정답 1개가 탈락. δ는 전 지표 선두 유지.

## 결론
1. **쿼리 단계 테스트 필터가 정답** — demotion은 composer에만 있어 그래프를 직접 호출하는 β/γ는 테스트가 샜다. 도구 응답에 `exclude_tests` 필터를 적용하니 재인덱싱 없이 모든 그래프 방식이 테스트오염 0/90 달성.
2. **β/γ에는 순이득** — 토큰 감소 + 관련성precision·효율 상승. 제거된 파일은 노이즈였음.
3. **δ(자동선별)는 여전히 전 지표 선두** — 정답존재 97%, 토큰 최소(4,800), 충분성 87%, 효율 18.06, 오염 0.
4. **β(그래프 전체 덤프)는 여전히 최약 그래프 방식** — 셋 중 토큰 최다·효율 최저이며 유일하게 정답존재가 하락. 무차별 폭은 필터 후에도 비쌈.
5. **α(grep)는 필터 미적용** — 15/90·최다 토큰 유지. 높은 관련성precision(0.776)은 파일을 적게 통째로 주기 때문.

## 한계 (해석 주의)
- gov-validator: δ는 glossary로 100% 커버하나 β/γ는 여전히 0%(semantic_search→확장 경로로는 밸리데이터 제네시스 정답에 도달 못함, 테스트 필터와 무관).
- `exclude_tests`는 opt-in — 테스트가 필요한 의도("테스트 추가")는 생략하면 됨. 벤치마크 β/γ는 설계/구현 질의를 모사하려 opt-in.
- 도메인별 수치는 문항 1~3개 기준(경향만). 검색 3회 평균 / 판정 3회 전부(방식당 90셀)·단일 투표.
