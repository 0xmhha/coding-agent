# [CKG] 그래프 기반 코드 검색 효과 측정 — 4-Way 비교 결과 (v5)

## 배경 / 목적
Code Knowledge Graph(ckg)가 AI 코드 이해에 실제로 효과가 있는지를 **느낌이 아닌 수치**로 검증한다.
미리 정답을 아는 **30개 코드 질문**(go-stablenet 12개 도메인)을 **4가지 컨텍스트 제공 방식**으로 제출하고 응답 품질을 자동 측정·비교했다.

- 인덱스: go-stablenet `dev`(`c051d50b`) 기준, **`make gstable` 빌드 파일만** 스코핑(미사용 합의엔진 제외, Solidity 컨트랙트 포함), 프로덕션 우선 랭킹 적용
- 검색 3회 평균 / LLM 판정(Sonnet, PARSE_FAIL 0) / 입력은 키워드·문장만(정답은 채점 전용 — 누수 없음)
- **v5 변경점**: δ(get_for_task) 경로에 **glossary 기반 쿼리 확장** 적용. 도메인 개념→심볼 매핑을 추가해 키워드만으로는 닿지 못하던 정답을 의미검색이 회수하도록 함. 매핑은 도메인 지식 엔트리에 저장되어 `cks-glossary-gen`이 **DB 재생성 시 자동 재생성** → 효과 영속.

## 비교 방식
| 코드 | 방식 |
|------|------|
| α | 파일 원문 grep (기준선, 그래프 미사용) |
| β | 그래프 전체 + 노드 본문 |
| γ | 필요한 정보 개별 조회 + 노드 본문 |
| δ | 자동 선별 (get_for_task, glossary 확장) |

## 핵심 결과
| 방식 | 정답존재 | 정답집중도 | 관련성precision | 설계충분성 | 평균토큰 | 효율(충분성/1k토큰) | 테스트오염 |
|------|------:|----------:|----------:|----------:|--------:|------:|----------:|
| α grep | 70% | 0.266 | 0.756 | 80% | 10,829 | 7.39 | 15/90 |
| β 그래프전체 | 83% | 0.034 | 0.293 | 80% | 11,811 | 6.77 | 78/90 |
| γ 개별조회 | 83% | 0.061 | 0.372 | 80% | 7,056 | 11.34 | 69/90 |
| **δ 자동선별** | **97%** | 0.263 | **0.641** | **87%** | **4,767** | **18.18** | **3/90** |

- **정답존재**(위치 정확도): 정답 파일을 검색 결과에 포함했는지 — "정답을 실제로 가져왔는가"
- **정답집중도**(strict): 가져온 파일 중 정확히 정답 파일의 비율(관련 맥락은 모두 off-target 처리되어 낮아짐)
- **관련성precision**(LLM 판정): 정답 + 설계에 실제로 유용한 맥락(호출자/피호출자/타입/관련 컨트랙트)의 비율 — 그래프가 가져온 유용한 주변 코드를 인정
- **설계충분성**(LLM 판정): 그 컨텍스트로 기능 수정/설계가 가능한 수준인지
- **테스트오염**: 검색 결과에 테스트 파일이 1개 이상 섞인 셀 수(90 = 30문항×3회 중)

## glossary 쿼리 확장 효과 (δ, v4→v5)
| δ 지표 | v4(확장 전) | v5(확장 후) | Δ |
|--------|--------:|--------:|---|
| 정답존재 | 83% | **97%** | +14pp |
| 정답집중도 | 0.208 | 0.263 | +0.055 |
| 관련성precision | 0.594 | 0.641 | +0.047 |
| 설계충분성 | 77% | **87%** | +10pp |
| 평균토큰 | 5,398 | 4,767 | −631 |
| 효율 | 14.20 | **18.18** | +3.98 |

모든 그래프 방식에서 공통으로 놓치던 4문항(밸리데이터셋 생성·GovValidator 제네시스 초기화·스토리지 슬롯 레이아웃·민터 발행 한도)은 **그래프 구축 누락이 아니라** 의미검색이 정답을 컷오프 밖으로 랭킹한 것이 원인이었다(파일은 인덱싱되어 있었음). glossary에 해당 도메인의 개념→심볼 매핑이 없었기 때문. **검증된 도메인 지식 엔트리 4개**를 추가해 해소 → δ 정답존재 83%→97%, 충분성·효율↑, 토큰↓. α/β/γ는 grep / raw 의미검색이라 composer 확장을 우회 → 불변.

## strict vs 관련성 precision (분리의 의미)
| 방식 | 정답집중도(strict) | 관련성precision | 해석 |
|------|----------:|----------:|------|
| α grep | 0.266 | 0.756 | 파일 수 적고 대부분 관련 |
| β 그래프전체 | 0.034 | 0.293 | 정확한 정답보다 ~9배 많은 파일이 관련 맥락 |
| γ 개별조회 | 0.061 | 0.372 | 그래프 폭은 대부분 유용한 맥락(노이즈 아님) |
| δ 자동선별 | 0.263 | 0.641 | 간결한 팩, ~64% 직접 관련 |

strict 지표만 보면 그래프 방식이 96~97% 노이즈로 보이나(β 0.034, γ 0.061), 파일별 관련성을 판정하면 β/γ가 가져온 것의 29~37%가 설계에 실제로 유용하다. 그래프의 폭은 대체로 유용한 맥락이며, 정답존재를 별도 지표로 두어 "관련성 높음"이 정답 누락을 가리지 못하게 했다.

## 결론
1. **그래프 자동 선별(δ)의 가치가 수치로 입증됨** — 정답존재 97%(최고, +14pp), 토큰 최소(4,767, grep의 약 1/2), 설계충분성 최고(87%), 효율 최고(18.18 — grep 기준선의 약 2.5배).
2. **glossary 쿼리 확장은 영속적·저비용 레버** — 도메인 엔트리 4개로 δ 정답존재 +14pp(토큰은 오히려 감소). DB 재생성 시 `cks-glossary-gen`이 재생성하므로 효과 유지.
3. **"그래프 전체 덤프(β)"는 역효과** — 토큰 최다(11,811)·효율 최저(6.77)·테스트오염 최다(78/90). 무차별 제공은 비효율.
4. **프로덕션 우선 랭킹이 핵심** — δ는 테스트오염 3/90으로 최저(β 78/90, γ 69/90는 composer 우회로 높음). β/γ가 추가 토큰을 지불하는 대상이 바로 이 테스트 맥락.
5. **α(grep)는 강하지만 비쌈** — 관련성precision 최고(0.756, 파일 전체)이나 정답존재 최저(70%, grep 한계)에 토큰 약 2.3배.

## 한계 (해석 주의)
- v4에서 지적된 **gov-validator 검색 품질 갭은 δ에서 해소**(밸리데이터셋/GovValidator 제네시스 문항 회수). β/γ는 composer 우회로 미해당.
- 도메인별 수치는 문항 1~3개 기준이라 **경향만** 의미(종합 수치가 안정적).
- 검색 3회 평균 / 판정 셀당 1회 — 노이즈 감소했으나 완전 제거는 아님(α/β/γ 관련성precision은 검색 변화 없이 v4→v5 ≤0.012 변동).
