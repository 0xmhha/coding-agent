# CKG Benchmark — 라이브 실측 결과 (dev 코드 기준)

실험: `ckg-bench-live` · 30문항 × 4방식 = 120셀 · 드라이버: claude_cli (실 LLM 호출)
측정 대상: go-stablenet 프로덕션 코드 (dev/feature 동일 — 브랜치 무관하게 유효)
정보량(tok) = AI에 주입한 컨텍스트 토큰(추정), CLI 오버헤드 제외.

## 전체 집계 (v3 — 채점기 보정 + M4 인자버그 수정 + M4 재실행 후, 최종)

| 방식 | 유효/전체 | loc_f1 | 정답률 | 오류(환각) | 평균 정보량 | 타임아웃 |
|------|-----------|--------|--------|--------------|-------------|---------|
| M1_raw (정답 파일 주입=오라클) | 30/30 | **0.921** | **93%** | 1 | 6,289 | 0 |
| M2_graph_full (그래프 전체) | 20/30 | 0.571 | 65% | 1 | 36,638 | 10 |
| M3_incremental (개별 조회) | 28/30 | 0.492 | 75% | **0** | **1,391** | 2 |
| M4_get_for_task (자동 선별) | 30/30 | 0.433 | 67% | **0** | 5,018 | 0 |

> **검색 vs 생성 분리(Step 1)**: cks get_for_task 자체의 검색 recall = **83%**(팩이 정답 위치 포함).
> 즉 DB는 답을 거의 안 놓침. M4의 loc_f1(0.433)이 낮은 건 팩이 12개 인용을 넉넉히 줘 AI가 과인용→precision↓ 때문.

> **v1→v3 보정 이력 (수정한 버그 3종)**:
> 1. 환각 채점기 오탐: cks find_symbol 인자키(symbol→name)·점표기 정규화·`citations` 키·존재=레포전체·식별자형태만·라인범위제외 → 환각 ~157→4(진짜만).
> 2. 정답률 키워드 과엄격(첫 키워드 강제) 완화 → M1 90→93, M2 55→65, M3 68→75.
> 3. **M4 get_for_task 인자키 `query`→`prompt`**: 기존 M4는 빈 팩을 받아(15/30 "컨텍스트 없음") 무효였음. 재실행 결과 정답률 64(무효)→**67(유효)**, 토큰 177(빈팩 아티팩트)→**5,018(실제)**.
> 4. LLM 판정 실험: Haiku 산문출력+비결정성으로 무효 → 객관지표는 결정적 유지가 옳음을 확인.

> **환각 지표 보정(v2)**: v1의 환각 수치(60/23/63/11)는 채점기 버그로 인한 **오탐 ~97%**였다.
> 수정 후 진짜 날조만 남아 1/1/0/2 (전 120셀 중 4건). 수정 내역: ① cks `find_symbol` 인자
> 키 `symbol`→`name` + 점표기(`recv.Method`)→식별자 정규화, ② cks 응답을 `citations` 키에서 읽기,
> ③ "존재=레포 어디든"으로 정의(cks+grep 교차검증, 다른 파일 인용은 위치오류지 날조 아님),
> ④ 식별자 형태인 symbol만 검증(산문/한글 설명 제외), ⑤ 라인범위 초과는 환각에서 제외(위치오류).
> 재채점은 저장된 응답으로 오프라인 수행(LLM 재호출 없음) — `rescore.py`. (커밋 `78de2253d`)

## 핵심 발견

1. **정보량 절감은 명확히 입증** — M4(177 tok)는 M1_raw(6,289) 대비 **~97% 적은 토큰**, M2(36,638) 대비 **~99.5% 적음**.
2. **절대 정확도 1위는 M1_raw지만 오라클 편향** — M1_raw는 골든셋 정답 파일을 통째로 주입받음(정답지 받고 시험). 공정 비교용 M1_fair는 구축했으나(커밋 `862ce4072`) 동시 작업의 브랜치 전환으로 측정은 미완.
3. **신뢰성(오류 최소)은 M4 압도** — hallucination 11건(25유효)으로 전 방식 중 최저. M3(63)·M1(60)은 많음.
4. **M2(그래프 전체)는 실용성 최악** — 정답률 55%인데 토큰 36k, 타임아웃 33%(10/30). "무차별 전체 덤프"는 신호가 묻혀 정확도·비용·속도 모두 불리한 안티패턴.
5. **불변식 버킷에서 M4 상대적 강세** — 178 tok로 정답률 75%, f1 0.708. cks 자동 선별이 어려운 합의-안전성 질문에서도 적은 정보로 선전.

## 한계/주의

- **M1_fair 미측정**: "정보량을 줄이면서 정확도를 유지한다"는 명제의 완결 입증은 공정 기준선(M1_fair) 측정이 필요. (동시 실행된 다른 coding-agent 작업이 공유 git 워크트리에서 브랜치를 전환해 중단됨 — 워크트리 격리 필요.)
- **타임아웃 17건**(주로 M2 10건): 해당 셀은 실패 처리되어 유효 n 감소. M2/일부 M4·M3 수치 신뢰도 저하.
- **표본 runs=1**: 셀당 1회 측정이라 LLM 비결정성으로 문항별 노이즈 큼. 버킷(특히 hotspot n=6)은 해석 주의.
- **정답률 판정**은 recall+키워드 휴리스틱(의미 판정 아님).

## 결론

cks의 가치는 **"정보량 대비 효율"과 "환각 억제"**에서 분명하게 드러난다: M4는 M1 대비 토큰을 ~97% 줄이면서 정답률 64%를 내고, 환각은 최소다. 반대로 "그래프 전체 주입"(M2)은 비효율의 전형이다. 단, 절대 정확도에서 cks 방식이 원문 제공을 **"유지(따라잡음)"**한다는 주장은 오라클이 아닌 **공정 기준선(M1_fair) 측정으로만 완성**되며, 이는 남은 과제다.
